Malicious npm Releases Hit ImmobiliareLabs Backstage GitLab and LDAP Plugins

A reported wave in the Miasma Mini Shai-Hulud campaign republished multiple @immobiliarelabs Backstage plugin versions with credential-stealing payloads, raising exposure concerns for developer portals and CI/CD environments.

Socket

What happened

Socket reported that attackers published malicious npm versions under the legitimate @immobiliarelabs scope on June 26, 2026, affecting Backstage plugins for GitLab integration and LDAP authentication. The activity is described as part of the broader Miasma Mini Shai-Hulud supply-chain campaign, which has focused on compromising trusted developer infrastructure, pushing tainted package releases, and stealing secrets from developer and CI/CD environments. The exact initial compromise path is still not fully confirmed; Socket noted GitHub Actions deployment activity and a possible upstream link involving codfish/semantic-release-action, but characterized the investigation as ongoing.

Key findings

  • The affected packages are legitimate npm packages under the @immobiliarelabs scope, not typosquats.
  • Socket identified malicious releases across multiple plugin families: @immobiliarelabs/backstage-plugin-gitlab, @immobiliarelabs/backstage-plugin-gitlab-backend, @immobiliarelabs/backstage-plugin-ldap-auth, and @immobiliarelabs/backstage-plugin-ldap-auth-backend.
  • The reported malicious versions span several major release lines, suggesting attackers attempted to reach users pinned to older versions as well as those consuming the latest release.
  • The payload reportedly stages JavaScript malware using Bun and targets developer credentials and CI/CD secrets.
  • The packages are relevant to Backstage environments, especially internal developer portals with GitLab and LDAP authentication integrations.
  • The initial access vector and full blast radius remain uncertain based on the available source.

Affected ecosystems

npm, Backstage plugin ecosystem, GitLab-integrated developer portals, LDAP authentication integrations, GitHub Actions / CI/CD environments

Security analysis

This incident is notable because the malicious releases targeted packages that can sit close to sensitive internal systems: Backstage developer portals, source-control integrations, and authentication workflows. A compromised frontend or backend plugin in this context may run in environments with access to service credentials, source repository tokens, internal APIs, or identity infrastructure. The reported republishing of multiple historical versions is also important for exposure analysis: organizations cannot limit checks to users tracking latest releases. Teams should inspect lockfiles and artifact repositories for exact installed versions and installation timestamps. The source attributes this to the ongoing Miasma Mini Shai-Hulud campaign, but some details remain unresolved, including the confirmed initial compromise path and whether all downstream users of the listed versions executed the payload. Defensive response should therefore combine package inventory review with credential exposure handling for any environment that installed or built the affected artifacts.

Defender actions

  • Search dependency manifests, lockfiles, SBOMs, artifact caches, and build logs for the affected @immobiliarelabs packages and versions published around June 26, 2026.
  • Prioritize review of Backstage deployments that include GitLab or LDAP authentication plugins from the @immobiliarelabs scope.
  • If a malicious version was installed or built, treat developer tokens, npm tokens, GitHub/GitLab credentials, CI/CD secrets, and cloud credentials available to that environment as potentially exposed.
  • Rotate secrets used by affected build jobs, Backstage deployments, release automation, and package publishing workflows.
  • Review CI logs for unexpected Bun installation or execution, outbound network activity, repository creation, token access, or unusual package publish events.
  • Pin to known-good package versions only after verifying package integrity from trusted sources; do not assume older major versions are safe because several historical release lines were reportedly republished.
  • Audit GitHub Actions and semantic-release workflows for unexpected dependency changes, compromised tokens, or unauthorized release activity.
  • Monitor npm advisory data and Socket updates for confirmed IOCs and any additional packages added to the campaign.
