Poisoned VS Code Extension Shows Editor Marketplaces Are a Software Supply-Chain Control Point

A malicious Nx Console release reportedly reached both VS Code Marketplace and Open VSX and was tied to theft of GitHub-internal repositories, highlighting the risk of trusted developer extensions.

Socket

What happened

Socket’s post is primarily a product announcement, but it cites a concrete supply-chain incident: GitHub disclosed in May 2026 that a poisoned third-party VS Code extension compromised an employee device and enabled exfiltration of roughly 3,800 internal repositories. The cited extension was Nx Console 18.95.0, described as a malicious release that appeared in both the Visual Studio Marketplace and Open VSX before being removed. The same source also references recurring GlassWorm activity abusing Open VSX with malicious or compromised extensions intended to steal developer credentials and propagate through editor marketplaces.

Key findings

  • The affected supply-chain surface is not a traditional package manager but developer editor extensions, which run inside workstations that often hold source code, secrets, terminals, and authenticated internal access.
  • Nx Console 18.95.0 is identified by the source as a malicious extension release that reached both VS Code Marketplace and Open VSX before removal.
  • The reported GitHub impact was significant: compromise of an employee device and exfiltration of approximately 3,800 internal repositories.
  • Open VSX exposure matters beyond VSCodium; the source notes use by VS Code-derived and AI-oriented editors such as Cursor, Windsurf, Antigravity, and others.
  • Details remain incomplete from the provided source: it does not include indicators of compromise, the exact malicious behavior in Nx Console 18.95.0, how many non-GitHub users installed it, or whether related packages/extensions were affected.

Affected ecosystems

Visual Studio Code Marketplace, Open VSX, VS Code-compatible editors and forks, Developer workstations and build/development environments with installed editor extensions

Security analysis

This incident reinforces that editor extensions should be treated as executable dependencies in the software supply chain. They are often installed by individual developers, updated automatically, and trusted because they come from familiar marketplaces or known project names. Once installed, an extension can operate in a high-value context: the developer’s local repositories, shell environment, credentials, cloud sessions, and internal network access. That makes extension compromise a practical path to source-code theft and downstream supply-chain intrusion. The Nx Console 18.95.0 case is especially notable because the source says the malicious release appeared in both major extension ecosystems before removal. That reduces the effectiveness of marketplace choice as a risk boundary: organizations using VS Code, Open VSX-based editors, or multiple AI-centric coding tools may have the same extension risk distributed across different registries. Because the provided source lacks technical indicators, defenders should focus first on exposure assessment and credential containment rather than assuming a complete detection recipe exists. The highest-value control is preventing unreviewed extensions from installing or updating in the first place, backed by centralized inventory, version pinning where practical, and rapid removal procedures for known-bad releases.

Defender actions

  • Audit current and historical installations of Nx Console, specifically version 18.95.0, across developer machines and remote development environments.
  • Review extension inventories for VS Code, VSCodium, Cursor, Windsurf, and other Open VSX-backed editors; do not limit checks to the standard VS Code Marketplace.
  • Correlate extension install or update times with unusual source-code access, token use, Git operations, terminal activity, and outbound network connections.
  • Rotate credentials and developer tokens exposed on systems where the malicious extension may have run, prioritizing repository, package registry, cloud, CI/CD, and SSO-linked credentials.
  • Implement allowlisting or approval workflows for editor extensions, especially extensions with broad filesystem, terminal, network, or language-server access.
  • Add install-time and update-time controls where possible; inventory-only approaches may identify exposure after execution has already occurred.
  • Monitor for repeated marketplace abuse such as renamed, republished, or compromised extensions that imitate trusted developer tooling.
