Self-replicating npm worm abuses binding.gyp in ai-sdk-ollama and other packages

Endor Labs reports an active npm campaign, dubbed Miasma, that trojanized ai-sdk-ollama and many other packages to execute during install, steal cloud and registry credentials, and spread through maintainer accounts.

Endor Labs

What happened

Endor Labs observed malicious npm publishes affecting ai-sdk-ollama, an unofficial bridge between Ollama and the Vercel AI SDK, along with dozens of other npm packages. The reported campaign uses npm’s native build path rather than an obvious lifecycle script: a malicious binding.gyp triggers node-gyp during installation and runs a payload before the package is imported by application code. The final payload is described as a multi-cloud credential stealer with self-propagation behavior, targeting cloud, Kubernetes, Vault, CI, and package registry secrets. The public source provides strong detail on technique and intent, but the complete affected package/version set appears broader than the excerpted list, so defenders should treat this as an active npm ecosystem incident rather than a single-package compromise.

Key findings

  • Four malicious ai-sdk-ollama versions were reportedly published to npm within a 17-second window; the legitimate library code was not the primary change, but additional files in the package tarball introduced the payload.
  • The execution trigger was binding.gyp, causing npm/node-gyp build behavior to run attacker code during install even where teams primarily monitor postinstall or other declared lifecycle scripts.
  • The malware used layered obfuscation and changed decoding parameters between versions, suggesting active evasion rather than accidental packaging artifacts.
  • The later-stage payload reportedly downloaded the Bun runtime and attempted to collect AWS, GCP, Azure, Vault, Kubernetes, GitHub Actions OIDC, npm, GitHub, and RubyGems credentials.
  • Endor Labs characterizes the activity as part of a self-replicating npm worm spreading across maintainer accounts, with ai-sdk-ollama representing one node in a larger campaign.

Affected ecosystems

npm / Node.js packages, especially installs that allow native builds via node-gyp, ai-sdk-ollama users, including projects integrating local Ollama models with the Vercel AI SDK through the unofficial package, Maintainer accounts and CI/CD environments with npm publish rights or registry tokens, Cloud and platform environments exposing AWS, GCP, Azure, Vault, Kubernetes, GitHub Actions OIDC, npm, GitHub, or RubyGems credentials during dependency installation, Other npm packages named in the source include @vapi-ai/server-sdk, @evolvconsulting/evolv-coder-lite, executable-stories-cypress, executable-stories-react, multiple autotel packages, and multiple awaitly packages; the excerpted source does not establish that this is the full affected set.

Security analysis

This incident is notable because it abuses a less commonly monitored npm execution path. Many supply-chain controls focus on package lifecycle scripts such as postinstall, but npm can also invoke node-gyp automatically when a package includes binding.gyp. In this case, that build mechanism became the install-time execution primitive. For AppSec teams, the practical lesson is that “no install scripts” is not equivalent to “no install-time code execution.” Native build metadata, package tarball diffs, and build-tool invocation should be part of dependency risk review. The credential targets also raise the impact beyond application dependency compromise. If the package installed in CI or on a maintainer machine, the attacker may have had access to cloud credentials, registry publish tokens, GitHub automation credentials, and Kubernetes or Vault material. That makes downstream propagation and infrastructure access plausible even if the malicious package was never imported by an application. The source indicates a broader worm campaign but provides only a partial view in the supplied excerpt. Defenders should therefore avoid scoping solely to ai-sdk-ollama. Search across npm dependency telemetry, package manager logs, and registry mirrors for the named package families and for anomalous binding.gyp additions in recently updated dependencies.

Defender actions

  • Identify all installations of ai-sdk-ollama and the other named npm packages during the reported June 3–4, 2026 window, including developer workstations, CI runners, build containers, and cached package layers.
  • Inspect lockfiles, npm cache, artifact repositories, and SBOMs for malicious versions. Because tarball contents can differ from repository source, compare installed package artifacts rather than relying only on upstream Git history.
  • Add detection for unexpected binding.gyp files and native build execution in packages that do not normally require native compilation. Do not rely only on postinstall or lifecycle-script scanning.
  • If exposure is possible, rotate potentially accessible secrets: npm tokens, GitHub tokens, GitHub Actions OIDC trust assumptions, cloud credentials, Vault tokens, Kubernetes credentials, and any RubyGems credentials present in build or developer environments.
  • Review npm maintainer accounts for unauthorized publishes, new access tokens, suspicious 2FA events, and unexpected package version releases.
  • Rebuild affected artifacts from clean environments after dependency removal or pinning to known-good versions. Treat builds performed during the exposure window as potentially contaminated.
  • Restrict dependency install privileges in CI: use ephemeral runners, minimize environment secrets available during install, prefer read-only registry tokens, and separate dependency resolution from privileged deployment steps.
