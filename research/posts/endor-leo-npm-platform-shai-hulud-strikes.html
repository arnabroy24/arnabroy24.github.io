Stolen npm token used to poison 23 Leo Platform packages

A June 24 npm compromise pushed malicious Leo Platform releases that abused native build configuration rather than package lifecycle scripts, redirected at least one dist-tag, and attempted credential theft and further propagation.

Endor Labs

What happened

Endor Labs reports that an attacker used a compromised npm maintainer token for the Leo Platform JavaScript ecosystem to publish 23 malicious package versions on June 24, 2026. The packages were published within seconds under the maintainer account czirker and included a large obfuscated payload placed in index.js. Instead of relying on obvious npm lifecycle scripts, the malware was triggered through a binding.gyp native build path and could also execute when required by an application. The campaign targeted developer and CI/CD secrets, including cloud credentials, GitHub-related secrets, password manager data, Anthropic API keys, and other package registry credentials. The source also reports that the attacker manipulated npm dist-tags, including making leo-sdk resolve to a malicious 6.0.19 release via latest while clean 7.x versions remained available under a different tag.

Key findings

  • The affected npm account reportedly published 23 poisoned Leo Platform-related packages in a six-second window on June 24, 2026.
  • The malware avoided package.json install scripts and instead used binding.gyp plus a replaced index.js payload, reducing the chance that simple lifecycle-script checks would catch it.
  • Endor Labs says the payload unpacked multiple secondary tools and attempted both credential theft and worm-like propagation through GitHub and other registries.
  • The reported dist-tag change is especially risky: projects installing leo-sdk without an exact version may have resolved to the malicious version despite newer clean 7.x releases existing.
  • Known affected versions include leo-sdk@6.0.19, leo-aws@2.0.4, leo-cli@3.0.3, leo-auth@4.0.6, multiple leo-connector packages, serverless-leo@3.0.14, serverless-convention@2.0.4, solo-nav@1.0.1, rstreams-metrics@2.0.2, rstreams-shard-util@1.0.1, leo-cdk-lib@0.0.2, and related release-candidate or beta connector packages.

Affected ecosystems

npm / JavaScript packages in the Leo Platform ecosystem, Node.js applications and CI jobs that installed affected versions after the malicious publish time, GitHub repositories and GitHub Actions environments reachable from infected developer or CI hosts, Cloud environments whose credentials were present on infected systems, Other package registries and developer tooling environments where harvested tokens may have been available

Security analysis

This incident has the hallmarks of a high-impact package ecosystem compromise: a legitimate maintainer credential was abused, multiple related packages were published nearly simultaneously, and npm metadata was changed so unpinned installs could select malicious code. The use of binding.gyp is a notable evasion technique because many dependency controls focus on preinstall/install/postinstall scripts and may miss native build execution paths. AppSec teams should treat exposure as broader than application runtime dependency use: a CI runner, developer workstation, or build container that merely installed one of the affected packages may have executed the payload. The source provides detailed package names and versions, but defenders should still validate current npm metadata independently because dist-tags and package availability can change after disclosure. The full victim scope is not stated in the source, so organizations should assume possible exposure only where affected packages were installed or built during the relevant window and then confirm through logs and lockfiles.

Defender actions

  • Search SBOMs, lockfiles, npm cache, CI logs, and artifact build records for the affected package names and versions, especially installs on or after 2026-06-24 23:04 UTC.
  • Do not rely only on package.json lifecycle-script detection; inspect native build files such as binding.gyp and unexpected large entry-point changes in dependencies.
  • Pin known-good versions explicitly and verify npm dist-tags before reinstalling Leo Platform packages. For leo-sdk, investigate any resolution to 6.0.19.
  • Rotate credentials exposed to systems that installed the packages, prioritizing npm tokens, GitHub tokens, cloud keys, Vault credentials, PyPI/RubyGems credentials, Anthropic API keys, and CI/CD secrets.
  • Review GitHub organizations for unexpected public repositories, suspicious commits, new or modified GitHub Actions workflows, and unauthorized repository secrets or deploy keys.
  • Inspect developer-assistant and IDE configuration paths mentioned by the source, including .claude/, .cursor/, .vscode/, and .github/copilot-instructions.md, for attacker-added hooks or instructions.
  • Rebuild affected artifacts from clean environments after dependency remediation and credential rotation.

Source notes

  • Primary source: Endor Labs, "Shai-Hulud Strikes Leo Platform npm," published June 25, 2026.
  • The report attributes the malicious publication to use of a stolen npm token for maintainer czirker and lists 23 affected package versions.
  • The available source is a single vendor analysis; independent confirmation and any package maintainer remediation status were not included in the provided snippets.
