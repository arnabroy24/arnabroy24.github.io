Miasma npm worm shifts to binding.gyp execution, compromising 57 packages

Semgrep reports a second Miasma wave affecting 57 npm packages and 286+ versions, using native build configuration to run malware during install and steal developer and CI credentials.

Semgrep

What happened

Semgrep describes a new wave of the Miasma self-propagating npm malware campaign. The reported v2 campaign compromised 57 npm packages across more than 286 malicious versions. Instead of relying on visible npm lifecycle scripts in package.json, the malicious packages use a small binding.gyp file that invokes node index.js during native build processing, causing code execution when the package is installed. The payload reportedly steals cloud and developer credentials, abuses GitHub infrastructure for exfiltration, and uses stolen publishing access to push additional malicious package versions.

Key findings

  • The affected ecosystem is npm, with Semgrep reporting 57 compromised packages and 286+ malicious versions.
  • The execution mechanism is notable: malicious behavior is triggered through binding.gyp rather than preinstall or postinstall scripts, so controls that only block or review lifecycle scripts may miss it.
  • The payload is reported to target AWS, GCP, Azure, GitHub Actions secrets, and local password stores including 1Password, gopass, and pass.
  • Semgrep says the malware attempts to extract GitHub Actions secrets from runner process memory via /proc/*/mem, increasing risk for CI environments that installed affected packages.
  • The campaign reportedly tampers with AI coding-assistant configuration files to create persistence or influence generated code.
  • The worm allegedly forges provenance attestations while publishing follow-on malicious versions, so provenance metadata alone should not be treated as proof that a package version is safe.

Affected ecosystems

npm packages and npm-based build pipelines, GitHub Actions environments that installed affected versions, Developer workstations with npm installs of affected versions, Cloud accounts exposed through stolen AWS, GCP, or Azure credentials, Password/secret stores accessible to the install process, AI coding-assistant configuration files on affected developer systems

Security analysis

This incident is significant because it moves npm malware execution away from the most commonly monitored location. Many organizations have added policy checks for install scripts after prior npm malware campaigns, but binding.gyp is a legitimate native addon build file and may be overlooked by allowlists, manual review, or simple package diffing. A package does not need to advertise a lifecycle hook in package.json for installation to become dangerous if the build toolchain evaluates command substitutions from native build metadata. The campaign also illustrates how package compromise becomes an identity compromise problem. If the malware runs in CI, it may inherit registry tokens, GitHub Actions secrets, cloud credentials, and deployment permissions. If it runs on a developer workstation, it may reach local password stores and configuration files. The reported use of stolen credentials to publish more malicious versions means defenders should investigate publisher account integrity, not just remove the dependency. The source provides a large affected-package list, but the supplied snippet appears incomplete relative to the stated total of 57 packages and 286+ versions. Teams should use Semgrep’s full advisory or package-registry takedown data as the authoritative package/version inventory before closing exposure assessments.

Defender actions

  • Search dependency manifests, lockfiles, package manager caches, artifact SBOMs, and CI logs for the affected packages and versions reported by Semgrep. Treat both direct and transitive dependency installs as relevant.
  • Do not limit detection to package.json lifecycle scripts. Inspect suspicious npm packages for binding.gyp entries that execute shell commands or invoke Node scripts during install.
  • For any system that installed a reported malicious version, rotate npm tokens, GitHub tokens, GitHub Actions secrets, cloud keys, and other credentials accessible to the user or CI runner at install time.
  • Review GitHub Actions runner logs and host telemetry for unusual access to /proc/*/mem, unexpected node execution during npm install, outbound traffic to GitHub repositories not owned by the organization, and creation of unexpected package versions.
  • Audit npm publisher accounts for anomalous logins, token use, new automation tokens, changed maintainers, and unexplained releases. Revoke stale automation credentials and enforce MFA where available.
  • Check developer workstations and build images for modified AI assistant configuration files, unexpected persistence, and credential-store access during the install window.
  • Pin known-good versions, clear dependency caches, rebuild images from trusted bases, and avoid reusing CI workspaces that may have executed affected packages.
  • Use provenance as one signal, not a sole allow decision, because this campaign is reported to manipulate attestations.

Source notes

  • Primary source: Semgrep, “Miasma v2: Self-Spreading npm Worm Now Uses Malicious binding.gyp file and Compromises 57 Packages,” published June 4, 2026.
