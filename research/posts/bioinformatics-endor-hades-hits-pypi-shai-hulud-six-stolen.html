Shai-Hulud resurfaces in PyPI bioinformatics packages through stolen publishing tokens

Six Python packages used in genomics and phenotype tooling were pushed with malicious releases that hid a credential-stealing payload inside compiled extensions.

Endor Labs

What happened

Endor Labs reported a coordinated PyPI compromise on June 8, 2026 affecting six bioinformatics and graph-analysis packages: ensmallen, embiggen, pyphetools, gpsea, phenopacket-store-toolkit, and ppkt2synergy. The malicious versions were uploaded within about a minute using publishing tokens rather than the projects’ normal release pipelines. The packages carried a Shai-Hulud credential-stealing worm, with execution embedded in native binary extensions that run when the Python module is imported. Endor says the releases were quarantined on PyPI after reporting, but organizations may still be exposed through local caches, mirrors, build artifacts, notebooks, or containers that installed the affected versions before removal.

Key findings

  • Affected PyPI releases: ensmallen 0.8.101, embiggen 0.11.97, pyphetools 0.9.120, gpsea 0.9.14, phenopacket-store-toolkit 0.1.7, and ppkt2synergy 0.1.1.
  • The releases reportedly had no matching upstream GitHub commits, tags, or project releases, indicating unauthorized registry-only publication.
  • All uploads used a Bun JavaScript runtime user agent rather than common Python packaging tools, suggesting direct token abuse through custom automation.
  • The payload was not limited to Python source changes; Endor reports malicious behavior was placed in compiled .abi3.so extensions and triggered during module import.
  • The campaign reused Shai-Hulud-style credential theft and worm behavior previously observed in npm and earlier PyPI incidents.
  • The source attributes the publication to stolen API tokens, but does not describe how those tokens were obtained.

Affected ecosystems

PyPI / Python packaging, Academic and commercial bioinformatics tooling, Genomics, phenotype-analysis, graph machine-learning, and knowledge-graph research environments, Downstream CI systems, notebooks, containers, package mirrors, and artifact caches that may have pulled the malicious versions

Security analysis

This incident is notable because it combines classic package registry account abuse with a harder-to-review execution path. Many dependency reviews focus on Python source diffs, setup scripts, or install-time hooks. In this case, the reported trigger sat in compiled native extensions that are normally loaded as part of legitimate package functionality. That raises the risk for scientific Python users, who often rely on binary wheels for performance and may execute packages inside high-trust research environments containing API credentials, datasets, and cloud access. The narrow package set suggests targeted compromise of maintainers or project publishing credentials rather than broad typosquatting, but the source material does not establish the initial token theft method. Because PyPI reportedly quarantined the releases quickly, the main exposure question is whether organizations installed them during the short availability window or cached them internally afterward.

Defender actions

  • Search dependency manifests, lockfiles, SBOMs, package-manager logs, internal PyPI mirrors, container images, and notebook environments for the six affected package versions.
  • Treat any environment that installed and imported one of the malicious versions as potentially credential-exposed; prioritize developer machines, CI runners, research workstations, and shared notebook servers.
  • Rotate secrets accessible from exposed hosts, including PyPI, GitHub, npm, cloud, SSH, database, and CI/CD tokens where applicable.
  • Remove the affected versions from internal package caches and block reinstallation through dependency policy controls.
  • Review runtime telemetry for unexpected JavaScript execution, Bun usage, unusual outbound network activity, or new files/processes spawned from Python workloads around the reported publication window.
  • For maintainers, revoke existing PyPI publishing tokens, migrate to PyPI Trusted Publishing where possible, enforce 2FA, scope tokens tightly, and monitor for releases that do not correspond to signed source-control tags.
  • Add package review checks for native extensions and large embedded scripts, especially when a release differs from the upstream repository or build provenance is missing.

Source notes

  • Primary source: Endor Labs report published June 8, 2026.
  • The source states the packages were quarantined through PyPI reporting; independent registry status was not provided in the supplied material.
  • Details about the full payload behavior, exfiltration endpoints, and token-theft origin are incomplete in the supplied snippet, so affected organizations should validate with their own telemetry.
