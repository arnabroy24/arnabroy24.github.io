Compromised Mastra npm Releases Pulled a Typosquatted Infostealer Dependency

More than 140 packages in the @mastra npm scope were reportedly republished with a malicious dependency that executed during installation, putting developer workstations and CI runners at risk before application code was ever run.

Socket

What happened

Socket reports that on June 17, 2026, a single npm account published malicious versions of 141 packages in the @mastra/* namespace. The package contents were reportedly unchanged from legitimate releases, but their manifests added a dependency on easy-day-js, a typosquat of the popular dayjs library. A later version of easy-day-js contained a postinstall script that downloaded and ran a second-stage payload. Socket describes the payload as a cross-platform infostealer targeting browser data and more than 160 cryptocurrency wallet browser extensions, with persistence mechanisms for Windows, macOS, and Linux.

Key findings

  • The campaign abused the npm ecosystem by modifying legitimate package manifests rather than visibly changing application code.
  • The malicious dependency, easy-day-js, was reportedly published clean first and then updated with a weaponized postinstall hook.
  • Execution occurred during npm install, so importing or using an affected Mastra package was not required for exposure.
  • Socket states that @mastra/core alone has more than 918,000 weekly npm downloads, suggesting a potentially large blast radius.
  • The exact number of compromised downstream systems is not yet known from the available source.

Affected ecosystems

npm, @mastra/* packages, including @mastra/core, Developer workstations running npm installs, CI/CD runners and build environments that installed affected versions, Windows, macOS, and Linux hosts targeted by the reported second-stage malware

Security analysis

This incident is a strong example of why dependency review cannot stop at package names or source diffs. According to Socket, the compromised Mastra releases were largely identical to prior legitimate builds; the malicious behavior entered through a newly added dependency. That makes lockfile review, provenance checks, publisher-account monitoring, and install-script controls especially important. The most important operational detail is timing of execution. Because the payload ran from a postinstall hook, exposure begins when the package manager resolves and installs the dependency tree. CI systems are particularly sensitive because they often contain deploy keys, registry tokens, cloud credentials, and signing material. Developer laptops are also high value because the reported malware targeted browser and cryptocurrency wallet data. The available reporting does not establish how the npm account was compromised, which exact downstream projects installed the bad versions, or the complete set of C2 indicators. Defenders should therefore scope from local package-manager evidence: package-lock.json, npm cache, artifact build logs, CI logs, and endpoint telemetry from the compromise window.

Defender actions

  • Inventory whether any @mastra/* packages were installed or built from versions published during the reported June 17, 2026 compromise window.
  • Treat any host or CI runner that installed affected versions as potentially compromised, even if the package was never imported by application code.
  • Remove affected versions, delete node_modules and lockfiles as appropriate, and reinstall from known-good versions after verification.
  • Search dependency manifests and lockfiles for easy-day-js, especially version 1.11.22 or later versions associated with the campaign.
  • Rotate credentials that may have been present on exposed systems, including npm tokens, Git credentials, SSH keys, cloud keys, CI secrets, and wallet-related secrets.
  • Review CI job logs and endpoint telemetry for unexpected node postinstall execution, outbound connections to unfamiliar infrastructure, detached background processes, or persistence changes.
  • Consider blocking install scripts by default in high-risk build contexts where feasible, and allow them only for reviewed dependencies.

Source notes

  • Socket reported the campaign and attributed the delivery mechanism to a typosquatted npm dependency named easy-day-js added to malicious @mastra/* releases.
  • Socket reported that 141 @mastra/* packages were published by the npm account ehindero between roughly 01:15 and 02:36 UTC on 2026-06-17.
  • Socket reported that easy-day-js was first published as a clean package and later updated with a postinstall payload.
