PolinRider campaign spreads beyond npm into Packagist, Go modules, and Chrome extensions

Socket reports 108 malicious packages or extensions tied to a North Korea-linked developer-targeting campaign that hides JavaScript loaders in legitimate repositories and release artifacts.

Socket

What happened

Socket’s research describes an active software supply-chain campaign, tracked as PolinRider, affecting multiple developer ecosystems. The reported activity includes 162 malicious release artifacts across 108 packages and extensions spanning npm, Packagist, Go modules, and a Chrome extension. The campaign is linked by Socket to the broader North Korea-associated Contagious Interview / Famous Chollima cluster, but attribution should be treated as vendor-assessed rather than independently confirmed here. The tradecraft centers on compromised or abused maintainer access, modified legitimate repositories, and malicious package releases that execute hidden JavaScript loaders in developer environments.

Key findings

  • Socket reports malicious artifacts across npm, 10 Packagist packages, 80 Go modules, and one Chrome extension, with the campaign still active.
  • Attackers allegedly modify legitimate repositories and, where they have registry permissions, publish infected versions to downstream package ecosystems.
  • Malicious code is hidden using techniques such as long whitespace padding, fake .woff2 font files, and obfuscated JavaScript loaders.
  • Execution may be triggered through developer tooling, including VS Code task configuration, making developer workstations a primary exposure point.
  • Socket says the loader retrieves encrypted follow-on payload material from public blockchain/RPC infrastructure including TRON, Aptos, and BNB Smart Chain services, then decrypts and executes it.
  • Observed payloads reportedly include DEV#POPPER and OmniStealer, but the loader design means other payloads may be delivered.
  • The source notes use of force pushes and backdated commits, so visible GitHub history alone may not reliably show when malicious changes occurred.

Affected ecosystems

npm, Packagist / Composer, Go modules, Chrome extensions, GitHub-hosted source repositories, Developer workstations and VS Code environments

Security analysis

This incident is notable because it combines several high-risk supply-chain patterns: legitimate repository modification, possible maintainer-account compromise, malicious package publishing, and developer-tool execution. The expansion from npm into Packagist, Go modules, and Chrome extensions increases the chance that organizations will miss exposure if they only monitor one package registry. The reported use of hidden loaders also makes static review harder, particularly when malicious code is buried in configuration-like files or padded beyond normal editor views. For AppSec teams, the immediate question is not just whether an application imported a named package, but whether any developer or CI environment fetched and executed an affected release artifact. Because the payloads target developer environments and may steal credentials, downstream risk can include source-code access, package publishing tokens, cloud keys, and CI/CD secrets. Socket’s report indicates the campaign is ongoing and that additional affected artifacts may emerge, so exposure assessment should be repeated as new indicators are confirmed.

Defender actions

  • Check dependency inventories, lockfiles, package manager caches, and CI logs against Socket’s live PolinRider indicators and any newly published package/version lists.
  • Do not rely only on repository landing pages or current commit history; review release metadata, registry publish events, GitHub audit logs, force-push events, and maintainer account activity where available.
  • Inspect repositories and installed packages for suspicious one-line JavaScript, extreme whitespace padding, unexpected .woff2 files, altered VS Code task files, and other new execution hooks in configuration files.
  • Treat systems that installed confirmed malicious versions as potentially compromised: preserve evidence, rebuild from known-good sources, and rotate secrets from a clean environment.
  • Audit developer endpoints for credential theft and malware execution, especially where browser wallets, SSH keys, npm/Packagist/GitHub tokens, cloud credentials, or CI secrets may have been accessible.
  • Review maintainer account security: enforce MFA, validate recovery email domains, remove stale registry tokens, and verify that GitHub and package registry permissions are still appropriate.
  • For Go, Composer, and npm projects, prefer reproducible builds and pinned versions; investigate unexpected dependency upgrades around the reported compromise windows.
